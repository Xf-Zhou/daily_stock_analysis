# DSA Web 全站视觉优化设计

- 日期：2026-07-10
- 状态：已实施并验证
- 类型：前端视觉与交互改进
- 影响范围：`apps/dsa-web/`、`docs/CHANGELOG.md`

## 1. 背景

DSA Web 已具备浅色/深色主题、通用组件、响应式导航和多类业务页面，但现有视觉语言缺少统一性：登录页偏赛博量化风，工作台页面偏深色数据终端，首页与列表页的信息密度和页面骨架也不一致。全局样式文件包含大量页面专用 token 与历史兼容样式，圆角、半透明表面、渐变、阴影和发光效果使用较多，削弱了信息层级。

本次优化以 [shadcn/ui](https://ui.shadcn.com/) 当前官网和 Dashboard Blocks 的视觉语言为参照：中性色、细边框、紧凑圆角、克制阴影、明确排版层级，以及由内容和组件状态承担视觉重点。

## 2. 目标

1. 让登录页、应用壳和所有主要业务页面呈现为同一套产品。
2. 默认使用清爽的浅色主题，同时完整保留深色主题切换。
3. 建立可复用的视觉 token 和组件规范，减少页面级样式漂移。
4. 提升导航可发现性、页面信息层级、表格可读性和移动端体验。
5. 保持现有 API、Schema、状态管理、路由和业务行为不变。

## 3. 非目标

- 不改变股票分析、候选评分、持仓、回测、问股或设置的业务逻辑。
- 不调整后端 API、数据结构、认证语义或任务状态协议。
- 不引入新的前端组件框架，也不执行 shadcn CLI 迁移；现有 React、Tailwind 和通用组件继续作为实现基础。
- 不重做信息架构，不新增页面或功能入口。
- 不借本次样式改造进行无关状态管理、API 层或构建系统重构。

## 4. 已确认的设计方向

### 4.1 主题与视觉语言

- 浅色主题为默认主题；深色主题继续可用。
- 主视觉使用中性黑、白、灰。现有青色和紫色不再承担大面积背景、渐变、阴影或导航强调。
- 主操作按钮使用高对比度的纯色前景/背景组合；次操作使用边框或幽灵样式。
- 红、绿、黄等颜色仅表达涨跌、成功、警告、危险和任务状态。
- 背景层级主要依赖颜色差异和边框，不依赖玻璃拟态或强阴影。
- 全站基础圆角收敛到 8–12px；胶囊形状仅用于 Badge、状态和分段控制器。
- 字体采用本地优先的现代无衬线字体栈，不加载远程字体资源。标题使用字重和紧凑字距建立层级，正文保持自然字距。

建议基础 token：

| 用途 | 浅色 | 深色 |
| --- | --- | --- |
| 页面背景 | 近白色 | 近黑色 |
| 主文字 | 近黑色 | 近白色 |
| 卡片背景 | 白色 | 深灰黑 |
| 次级表面 | 浅灰 | 深灰 |
| 边框 | 中性浅灰 | 中性深灰 |
| 主操作 | 黑底白字 | 白底黑字 |
| 涨/危险 | 红色语义色 | 红色语义色 |
| 跌/成功 | 绿色语义色 | 绿色语义色 |

具体色值通过现有 CSS HSL token 统一定义，不在页面组件中写死。

### 4.2 应用壳与导航

桌面端采用 shadcn Dashboard Block 风格的标准文字侧栏：

- 展开宽度约 240px，折叠宽度约 64–72px。
- 顶部显示 DSA 标识和产品名称。
- 导航按“主要功能 / 研究工具 / 系统”分组。
- 当前路由使用浅灰选中背景和清晰文字，不使用发光竖线。
- 主题切换、认证退出等低频操作放在侧栏底部。
- 中等宽度屏幕允许折叠成图标栏，并为图标提供 Tooltip。
- 移动端使用顶部菜单按钮和左侧 Drawer，主内容占满可用宽度。

主内容区使用统一最大宽度、页边距和纵向节奏。页面不再通过悬浮大圆角容器包裹整个壳层。

## 5. 组件设计

### 5.1 基础组件

统一以下组件的尺寸、颜色、圆角、焦点和禁用状态：

- `Button`
- `Input`
- `Select`
- `Checkbox`
- `Badge`
- `Card`
- `SectionCard`
- `StatCard`
- `PageHeader`
- `Toolbar`
- `InlineAlert` / `ApiErrorAlert`
- `EmptyState` / `Loading`
- `ConfirmDialog` / `Drawer`
- `Pagination`
- `Tooltip`

组件规范：

- 常规控件高度以 36–40px 为主，触控场景不低于 40px。
- 焦点状态使用清晰、统一的 ring；不通过发光阴影表达焦点。
- Card 默认使用细边框和纯色背景；仅弹窗、抽屉等浮层使用明显阴影。
- Badge 使用紧凑尺寸和低饱和背景，避免抢占正文视觉权重。
- 表单错误、接口错误和警告使用统一 Alert 结构，包含标题、说明和必要操作。

### 5.2 数据表格

- 表头使用次级文字和轻微背景差异。
- 行分隔使用细边框；悬停状态仅提供轻微背景变化。
- 数字列按内容需要右对齐，并使用 tabular numbers。
- 股票涨跌颜色继续遵循现有 A 股语义，不因主题改变含义。
- 行级操作收敛为主要操作加紧凑次操作，必要时进入更多菜单。
- 窄屏允许表格横向滚动，并在滚动容器上保留可访问的说明。

## 6. 页面设计

### 6.1 首页

- 保留“历史分析 + 报告工作区”的双栏核心结构。
- 顶部集中股票搜索、推送选项、大盘复盘和分析操作。
- 历史记录栏使用固定但可响应的宽度，减少大面积空白和强边框。
- 报告空状态、加载态和详情态共享同一内容画布。
- 移动端历史记录进入 Drawer，报告内容优先展示。

### 6.2 发现与候选

统一为以下页面顺序：

1. 简洁页面标题与必要操作。
2. 筛选工具栏。
3. 指标摘要或榜单状态。
4. 主数据表格。
5. 分页与每页数量。

候选评分、行情状态、行业标签和操作按钮沿用现有数据与行为，只调整视觉权重和排布。

### 6.3 持仓与回测

- 使用同一套 PageHeader、Toolbar、StatCard 和数据表格结构。
- 汇总指标放在顶部卡片中，明细和任务记录进入统一表格容器。
- 危险操作与强制操作保留明确确认和语义色，但避免大面积高饱和背景。

### 6.4 问股

- 保留会话栏、消息区和输入区的三部分结构。
- 会话栏遵循应用侧栏的视觉语言，但保持页面内从属层级。
- 消息内容优先，弱化气泡边框、头像发光和无必要装饰。
- 工具调用、思考、生成和完成状态使用紧凑的状态行。
- 输入区保持底部可见，并清晰区分主要发送操作和附加能力。

### 6.5 设置

- 保留分类导航和表单卡片。
- 统一字段标签、辅助说明、必填/敏感标识、保存操作和错误提示。
- 分类导航使用中性选中背景，不使用渐变或发光。
- 长表单使用清晰分组和稳定的操作区，避免多套表面 token 并存。

### 6.6 登录

- 移除粒子背景、强光晕、版本徽章和终端式安全文案。
- 使用简洁的居中或左右分栏表单布局，以品牌、标题、说明、认证字段和主按钮为核心。
- 密码初始化、MFA、错误反馈和重试行为保持不变。
- 登录页与工作台共享主题 token、输入组件和按钮规范。

## 7. 数据流与错误处理

本次不修改数据流：

- API 请求继续通过现有 `src/api/` 模块。
- 页面状态继续由现有 hooks、contexts 和 Zustand stores 管理。
- 路由和认证重定向逻辑保持不变。
- 任务流、SSE、轮询、缓存和 fallback 语义保持不变。

只统一状态呈现：

- 页面级失败使用 `ApiErrorAlert`。
- 局部请求失败使用 `InlineAlert`，不阻断其他可用区域。
- 首次加载使用 Skeleton 或已有 Loading 组件。
- 无数据使用 EmptyState，并提供下一步操作时才显示按钮。
- 降级数据、缓存数据和静态数据继续通过 Badge/Alert 明确标识。

## 8. 响应式与可访问性

- 大屏：展开文字侧栏，页面使用完整工具栏和表格。
- 中屏：侧栏可折叠，工具栏允许换行或分组。
- 小屏：Drawer 导航；页面标题、筛选和操作纵向排列；表格横向滚动。
- 所有交互元素保留可见焦点状态和键盘操作能力。
- 图标按钮必须提供 `aria-label`，折叠导航必须提供 Tooltip。
- 文本与背景满足合理对比度，语义不得仅依赖颜色。
- 动效集中在侧栏、Drawer、Dialog 和状态切换，时长以 150–200ms 为主。
- 遵循 `prefers-reduced-motion`，减少或关闭非必要动画。

### 8.1 超宽屏首页工作区

- 首页在 1440px 常规桌面宽度下保持现有密度和阅读节奏。
- 当可用主内容宽度超过 1440px 时，工作区逐级放宽；最大内容宽度设为 2160px，不使用无上限全宽布局。
- 历史分析栏保持约 288px 固定宽度，报告区域吸收新增空间；正文继续使用已有行宽约束，避免超长文本行。
- 情绪指标列保持合理比例，不因 2K/4K 屏幕扩展成大面积空卡片。
- 超宽屏仍保留少量对称页边距，使侧栏、历史栏和报告区形成一个完整工作台，而不是贴边铺满。

### 8.2 超宽屏发现与候选数据页

- 发现页和候选页在 2K/4K 屏幕下单独扩展到最大 2160px，不修改共享 `AppPage` 的 1440px 默认上限。
- 股票表格与候选表格使用新增宽度，降低无效留白并减少横向滚动；现有移动端最小表格宽度与滚动行为保持不变。
- 筛选工具栏不随页面无限拉伸：发现页搜索输入区域上限为 720px，候选页搜索输入区域上限为 720px，其余筛选项保持固定宽度。
- 发现页榜单仍保持四列上限，不因为超宽屏增加无意义的横向卡片数量；卡片内容继续使用紧凑层级。
- 其他阅读型页面继续使用共享 `AppPage` 的 1440px 上限，避免聊天、设置和说明文本过宽。

## 9. 样式与代码组织

- `index.css` 保留主题 token、基础样式和真正跨页面的组件样式。
- 新实现优先使用现有 Tailwind 工具类和通用组件，不新增平行设计系统。
- 页面迁移后删除确认无消费者的旧玻璃拟态、终端卡片、发光和页面专用兼容样式。
- 页面专用复杂样式仅在无法由组件或工具类清晰表达时保留，并使用明确命名。
- 不在组件中写死环境差异、端口、模型名或外部资源地址。

## 10. 验证方案

### 自动化验证

在 `apps/dsa-web/` 执行：

```bash
npm run test
npm run lint
npm run build
```

重点覆盖：

- ThemeProvider 和主题切换。
- Shell、SidebarNav 和移动端 Drawer。
- 基础组件的 variant、disabled、focus 和错误状态。
- 首页、发现、候选、问股、持仓、回测、设置和登录页现有测试。
- UI governance 测试需要随新规范同步更新，但不得降低有效约束。

### 视觉验证

- 1440px 桌面视口：浅色与深色。
- 390px 手机视口：浅色与深色。
- 检查首页、发现、候选、问股、持仓、回测、设置和登录页。
- 检查默认态、加载态、空状态、错误态、Dialog、Drawer 和长表格滚动。
- 使用 Playwright 截图进行人工对比，同时检查浏览器控制台错误。

## 11. 文档与变更记录

完成实现后更新 `docs/CHANGELOG.md` 的 `[Unreleased]`，使用扁平格式记录全站 Web 视觉与响应式改进。README 不更新，因为本次未改变项目定位、快速开始或主要入口。

### 11.1 实施结果（2026-07-10）

- 已完成浅色优先主题、通用组件、登录页、应用壳和主要业务页面的统一迁移。
- 桌面端侧栏已支持分组、折叠和展开；折叠后的路由、主题、退出与展开图标均提供鼠标悬停和键盘聚焦可见 Tooltip。
- 首页在超宽屏下采用最大 2160px 的自适应工作区；发现页和候选页采用页面级超宽布局，其他阅读型页面继续保持共享宽度上限。
- 登录和报告浏览器 smoke 已同步当前文案与稳定的 `data-slot` 选择器。
- 最终验证覆盖 500 个 Vitest、ESLint、TypeScript/Vite 生产构建，以及折叠侧栏 Chromium smoke。

## 12. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 全站样式改动引发局部回归 | 先迁移基础 token 和组件，再按页面逐步迁移并逐页验证 |
| 删除旧 CSS 影响隐藏状态 | 仅删除确认无消费者的样式；通过 `rg`、测试和视觉检查核对 |
| 浅色默认影响既有用户偏好 | ThemeProvider 继续尊重已保存的用户主题；默认值只作用于无历史偏好的用户 |
| 表格在移动端不可读 | 使用横向滚动和关键列优先，不强行将所有列压缩成卡片 |
| 页面功能在重排中丢失 | 不改数据流和事件处理；页面测试以现有行为为基线 |
| 超宽屏完全铺满导致卡片过度拉伸 | 首页最大宽度限制为 2160px，并保留正文行宽与固定历史栏 |
| 数据页全局放宽影响阅读型页面 | 仅在发现页和候选页覆盖到 2160px，共享 `AppPage` 保持 1440px |

## 13. 回滚方式

本次改动保持在 Web 前端样式、布局和通用组件范围内。若上线后出现严重视觉或交互回归，可整体回退本次前端改动；后端、API、Schema 和数据无需回滚。若只出现单页问题，可回退对应页面迁移，同时保留已验证的基础 token 和组件改进。
